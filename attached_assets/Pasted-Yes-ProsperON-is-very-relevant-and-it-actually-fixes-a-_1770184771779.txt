Candid risks ProsperON introduces (and how to preempt them)

Risk A: “Pay-to-win / influence capture”

If influence can be bought by burning, people will say it’s plutocratic.

Mitigation design you should state explicitly:
	•	decay already helps (freshness beats hoarding)
	•	add progressive caps or diminishing returns above a threshold (optional)
	•	create a “use-to-earn bias” so real usage mints more efficiently than pure support burn
	•	reserve a portion of pool allocation for usage-based routing (not just burn)

Risk B: “Regressive tax on poorer users”

If influence is tied to spending (fees/burn), it can disadvantage low-income users.

Mitigations:
	•	allow sponsored burns or builder boosts for onboarding
	•	incorporate participation-based credits for non-monetary contributions (careful, Sybil risk)
	•	keep “degraded mode UX” so lack of Prosper tier doesn’t block basic access

Risk C: “Yield pool optics and safety”

Even if “principal stays intact”, yield strategies have risk and complexity.

Mitigation:
	•	be explicit: “Season 0 is allocation accounting first”
	•	publish parameter dashboards + change logs (you already plan this)
	•	conservative strategy set initially, clear disclosures

Risk D: “Too many primitives at once”

If you pitch CHS + ProsperON + agent intent all in one breath, partners may hear “big bang”.

Mitigation: show phased plan:
	•	Phase 1: CHS + ProsperON integration (retention + allocation)
	•	Phase 2: agents
	•	Phase 3: governance experiments (optional)
