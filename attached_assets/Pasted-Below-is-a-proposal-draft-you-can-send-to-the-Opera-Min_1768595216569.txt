## 4) Best Practices Guidelines (Initial Draft)

Based on MiniPlay lessons learned, Zeno will adopt these standards across builds:

### Community & Brand Management

* Prefer a **broadcast channel** (updates, announcements, initiatives) over an open group for brand safety.
* Provide a **ticket/feedback system** (e.g., form + triage board) for issues and feature requests.
* Create structured community rituals: weekly challenges, patch notes, “what’s next” posts.

### Scalability from Day One

* Apps must withstand **hundreds of concurrent users** on launch day.
* Load testing, caching strategy, rate limits, and graceful degradation are mandatory.
* Observability: error tracking, latency monitoring, uptime alerts.

### Habit Loop / Daily Check-ins

* Include a **daily check-in** mechanic when it fits the app.
* Prefer check-ins that involve an **onchain transaction** (micro-action) to deepen wallet habit and onchain volume.
* Avoid direct “pay users to come back”; instead use progression, status, streaks, unlocks, and social proof.

### Roadmap + Release Rhythm

* Avoid shipping a massive feature set on day one.
* Launch small and improve continuously with a predictable cadence (weekly/biweekly drops).
* Roadmap should be visible enough to create anticipation without creating obligations.

### Analytics + Dashboards + Admin Controls

* Google Analytics (or equivalent) integrated by default.
* Internal KPI dashboard for the app + a MiniPay-wide dashboard view.
* Admin panel to tweak parameters safely without redeploying (limits, prices, difficulty, feature flags, etc.).

### Growth Loops (Referral + Social Proof)

* Referral mechanisms that reward *progress/status/unlocks* more than direct cash-like rewards.
* Social sharing prompts designed around achievement and identity (“look what I unlocked”), not spam.
* Viral hooks must be measurable: invite conversion, K-factor proxies, share rate.

### App Coin / Token Integration (Optional)

* Only when it meaningfully improves engagement or economic design.
* Must include anti-abuse and clear utility (unlocking, governance later if needed).
* Token design must avoid “infinite incentives” and volume farming loops.

## 5) KPI Framework (What We Measure)

Zeno will propose a standard set of KPIs across all MiniPay apps, with benchmarks and thresholds.

### Core KPIs (per app)

**Engagement**

* DAU / WAU / MAU
* Sessions per user per day
* Median session length
* Streak adoption rate

**Retention**

* D1, D7, D30 retention
* Cohort retention by acquisition channel

**Onchain Activity**

* Transactions per active user (daily/weekly)
* % of users completing at least 1 onchain action
* Repeat tx rate per user
* Fee volume (if applicable)

**Revenue**

* Revenue per active user
* Conversion rate to paid actions
* ARPDAU (or equivalent)
* Refund/dispute rate (if any)

**Growth**

* Invite/share rate
* Referral conversion rate
* Organic vs partner-driven adoption split

**Quality**

* Crash/error rate
* Latency (p95)
* Support ticket volume + resolution time

### Decision System: iterate, scale, delist

Zeno will create a scorecard and apply consistent actions:

* **Green**: scale (add features, invest in growth)
* **Yellow**: iterate (targeted fixes)
* **Red**: pause / delist candidate (unless clear fix exists)

This gives MiniPay an objective way to keep the ecosystem healthy.
